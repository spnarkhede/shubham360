# Contributing to the Cognitive Debt Prevention Kit

First — thank you. This kit improves when teams share what worked in practice, not just in theory.

---

## What We're Looking For

The best contributions come from real-world experience using these templates. Specifically:

- **Template improvements** — edits that made a template work better for your team
- **New real-world scenarios** — patterns or failure modes not yet covered
- **Simplifications** — anything that reduces friction without reducing value
- **Localization** — adapting templates to specific tech stacks, team structures, or industries
- **Bug fixes** — broken links, outdated references, contradictions between documents

We are not looking for:
- Adding AI tools to the "approved" list (teams define their own)
- Weakening the comprehension gates (the standards exist for a reason)
- Generic documentation improvements with no practical change

---

## How to Contribute

### Small changes (typos, links, minor wording)
Open a PR directly. No issue needed.

### Meaningful changes (template structure, new sections, new documents)
1. Open an issue first using the **Improvement Proposal** template
2. Describe what you changed, why, and what real-world problem it solved
3. Wait for acknowledgment before opening a PR — saves everyone time

### New documents
Open an issue with the **New Document Proposal** template. Include:
- What gap this fills
- Which existing documents it connects to
- A rough outline

---

## PR Guidelines

- Keep PRs focused on a single change or document
- Update the `README.md` document map if you add or rename files
- If your change affects the `DEFINITION_OF_DONE.md`, check it against `CODE_REVIEW_GUARDRAILS.md` for consistency
- Reference a concrete real-world scenario if your change is grounded in a specific project experience

---

## Tone and Style

These documents are written to be:
- **Direct** — no filler, no hedge language
- **Practical** — every section should be usable this week
- **Honest about tradeoffs** — acknowledge when something slows teams down and why it's worth it
- **Not preachy** — state the standard once, clearly; don't repeat it three ways

When in doubt, read the existing documents and match their register.

---

## Code of Conduct

Be direct, be constructive, be respectful. Disagreement is fine — dismissiveness isn't.

If you've shipped a system that accumulated cognitive debt and learned from it, that experience is valuable here. No judgment for how you got here.
